# Правила синхронизации взаимодействий персонажей

> Как правильно синхронизировать действия игроков с органами управления корабля.
> Выведено из разбора (декомпиляция ilspycmd) двух эталонных механик — **лебёдки**
> (`GPButtonRopeWinch`) и **штурвала** (`GPButtonSteeringWheel`). Применять при добавлении
> любой новой синхронизируемой механики.
>
> **Реализуешь взаимодействие — читай два файла вместе:**
> - **`GAME_INTERACTIONS.md`** — ЧТО делает игровой класс (поля, обработчики, авторитетное
>   состояние, класс политики). Это первая остановка: найди там свой тип, прежде чем декомпилировать.
> - **`INTERACTION_RULES.md`** (этот файл) — КАК это синхронизировать (правила R1–R16, выбор механизма).
>
> Также связано: `PLAN_SYNC.md` (что и в каком порядке синхронизировать), `Sync/InteractionPolicy.cs`
> (источник истины классификации SHARED/ITEM/LOCAL/HOST-ONLY), `CLAUDE.md` (координаты/фундамент F1–F5).

## Часть 1. Разбор двух эталонных механик

### 1.1. Лебёдка `GPButtonRopeWinch` — STATE-контрол

Поток (из `Update()`, пока игрок держит лебёдку — `stickyClickedBy`):

```
currentInput = keyboardDelta.y / dt * 0.69      // ввод = СКОРОСТЬ вращения
LimitInput(); ApplyWindResistance(); LimitTurning();
AddQuickReleaseInput();                          // клавиша Run → currentInput = -100 (быстрый сброс)
ApplyRotation();                                 // крутит трансформ ручки на currentInput*speed
GetRotationDelta();                              // дельта поворота ручки за кадр
DeltaToLength();   // rope.currentLength -= deltaRotation/360/gearRatio;  clamp 0..1
SendLengthToRope();// rope.changed = true
// не держат → currentInput = 0
```

Выводы:
- **Авторитетная переменная = `RopeController.currentLength`** (постоянное, ограниченное 0..1
  состояние). Ввод (скорость) интегрируется в неё.
- **Ручка лебёдки — косметика**: её поворот вычисляется из ввода, а длина — из поворота. На приёме
  достаточно знать `currentLength`; ручку можно не синхронизировать (или гнать как визуал).
- **Быстрый сброс — это НЕ отдельное событие**, а просто быстрое изменение `currentLength`
  (`currentInput=-100` → длина быстро уходит). Покрывается синком длины. Не нужен дискретный месседж.
- `RopeController` (база) хранит только `currentLength/currentResistance/changed`; подклассы
  (парус-угол/риф/якорь/штурвал-трос) переопределяют поведение, но состояние — та же длина.

### 1.2. Штурвал `GPButtonSteeringWheel` — COMMAND→PHYSICS-контрол

Поток (из `ExtraLateUpdate()`, пока держат):

```
currentInput += keyboardDelta.x * inputMult * 0.1   // ввод НАКАПЛИВАЕТСЯ в УГОЛ (команду)
ApplyRotationLimit();                                // clamp ±rotationAngleLimit
ApplyRudderRotation();   // attachedRudder.spring.targetPosition = limits.max * currentInput/rotationAngleLimit
ApplyWheelRotationFromRudder(); // визуал штурвала из rudder.currentAngle
// не держат → currentInput = RudderToWheelAngle(rudder.currentAngle)  // штурвал следует за рулём
```

Выводы:
- Лодка использует **старый `Rudder`** (HingeJoint `attachedRudder` + spring), НЕ
  `RopeControllerSteeringWheel`/`RudderNew`. Рулевого «троса» здесь нет — синк через `currentLength`
  не работает (это была ошибка, исправлено).
- **Авторитетная переменная = `currentInput`** (накопленная команда-угол), которая ставит
  **target пружины** руля-HingeJoint. **Угол руля — физический результат** интегрирования пружины.
- На кинематическом клиенте физика руля не воспроизводится → **угол руля у клиента нельзя считать
  локально**, его надо брать с хоста.

## Часть 2. Правила (R1–R10)

### R1. Синхронизируй АВТОРИТЕТНУЮ ПЕРЕМЕННУЮ, а не жест ввода
Не реплицируй сырой ввод (дельты мыши/клавиатуры) и не «проигрывай жест». Найди постоянную
переменную состояния, которую игра выводит из ввода, и синхронизируй ЕЁ.
Лебёдка → `currentLength`. Штурвал → `currentInput`.

### R2. Различай два типа контролов
- **STATE-контрол**: ввод пишет ограниченный постоянный скаляр напрямую (лебёдка→длина, якорный
  трос, риф). Детерминирован, одинаков у обоих при синке значения. → **Синкать значение состояния**
  (host↔client), с окном владения, пока игрок крутит.
- **COMMAND→PHYSICS-контрол**: ввод задаёт target, который физика интегрирует в расходящийся
  результат (штурвал→пружина→угол руля; угол гика от ветра на HingeJoint). → **Синкать КОМАНДУ
  хосту** (его физика — авторитет) И **отдельно гнать визуальный РЕЗУЛЬТАТ host→client**
  (кинематик-кукла / slerp поворота узла), т.к. физику клиент не повторит.

### R3. Хост — единственный авторитет физики
Корабль и физ-узлы у клиента кинематические (BoatSync/ControlsSync). Любой контрол, чей эффект идёт
через физику (руление меняет курс, авторитетный у BoatSync), ОБЯЗАН применяться на хосте. Клиент
только шлёт намерение и рисует результат хоста.

### R4. Окно владения — против «snap-back»
Пока игрок крутит STATE-контрол, НЕ применять эхо хоста для этого узла короткое время
(`LocalHoldMs≈600мс`), иначе локальная правка перетрётся до возврата round-trip.

### R5. Форвардить на изменение, throttled; финал — надёжно
Слать значение с фикс. частотой (ControlHz) только при изменении больше эпсилон; доставка
ReliableOrdered, чтобы финал (на отпускании) гарантированно дошёл. Не флудить, когда контрол держат
неподвижно.

### R6. «Рисуй, а не дерись» за локальный визуал
Не двигай трансформ контрола локально, если он узел, синхронизируемый с хоста. Пусть значение хоста
ведёт визуал; локальный ввод только форвардит команду.
> Это была ошибка штурвала: «освобождение» руля (un-kinematic) у клиента дралось с синком от хоста —
> руль переставал следовать. Исправление: руль всегда slerp'ится к хосту, клиент шлёт только
> `currentInput`.
> Исключение: контрол без host→client-визуала (ручка лебёдки) можно крутить локально, пока держат
> (не слёрпить его в этот момент).

### R7. Косметику/производные части не синкать
Ручка лебёдки, визуал-ротатор штурвала — вычисляются из авторитетного состояния, считаются локально.
Их синк — лишний трафик и источник «драк».

### R8. Дискретное vs непрерывное — проверь, не «быстрое непрерывное» ли это
- Непрерывные (кручение лебёдки/штурвала): стримить скаляр.
- Часть «дискретных» на деле — быстрое изменение непрерывного состояния (быстрый сброс лебёдки →
  резкий `currentLength`), отдельное событие НЕ нужно.
- Истинно дискретные изменения состояния (пришвартован/отвязан, переключатель крепления) → зеркало
  состояния или событие client→host→broadcast.

### R9. Сначала классифицируй по политике (InteractionPolicy)
{SHARED, ITEM, LOCAL, HOST-ONLY}. Синкать только SHARED; HOST-ONLY блокировать у клиента;
LOCAL/ITEM пропускать (личное/предметы — отдельный слой P3). Незнакомые палубные кнопки → SHARED.

### R10. Триггер бери из декомпиляции обработчика, а не из догадки
Реальные триггеры удивляют: расшвартовка = `OnPickup()` (не `OnAltActivate`); быстрый сброс = клавиша
`Run` во время удержания лебёдки; штурвал = `currentInput`, а не рулевой трос. Перед синком —
`ilspycmd -t ТипКнопки` и читать фактический обработчик/`Update`.

## Часть 3. Чек-лист для новой синхронизируемой механики

1. **Найти тип в `GAME_INTERACTIONS.md`** (обработчик, авторитетная переменная, класс политики). Если
   типа там нет — **декомпилировать** (`ilspycmd -t Имя`, либо всю сборку разом: `ilspycmd
   Assembly-CSharp.dll -o <dir>` → grep по `Assembly-CSharp.decompiled.cs`), понять, КАКАЯ переменная
   меняется в `OnActivate`/`Update`/`ExtraLateUpdate`/`OnScroll` (R10), и **дописать резюме в
   `GAME_INTERACTIONS.md`** (поведение + сетевой вывод, без полного кода).
2. **Классифицировать** (R9): SHARED / ITEM / LOCAL / HOST-ONLY. Если не SHARED — занести в
   `InteractionPolicy` и не синкать (или блокировать).
3. **Определить тип** (R2): STATE или COMMAND→PHYSICS?
   - STATE → синк значения host↔client, окно владения (R4), форвард-на-изменение (R5).
   - COMMAND→PHYSICS → форвард команды client→host (host применяет через игровой метод), визуал
     результата host→client отдельно (узел/кукла) (R3, R6).
4. **Косметику** (производные трансформы) НЕ синкать (R7).
5. **Кадр координат** для пspace-объектов: boat-local для стоящего на палубе, real-space для мира
   (см. `CLAUDE.md`; пример — `AnchorSync` с переключением кадра).
6. **Версия протокола**: добавить msg (append-only в `MsgType`), поднять `Protocol.Version`,
   зарегистрировать в `Protocol.Create`.
7. **Диагностика**: строка в `DebugOverlay` + `LogInfo` на форвард/применение — без этого баг не
   локализовать (тестировать только в игре, две копии).
8. **Очистка**: сбрасывать состояние в `Clear()` и на disconnect; восстанавливать изменённую физику
   (kinematic) при отключении.

## Часть 5. Паттерн «реле-через-хост по перехвату методов» (на примере швартовки)

Когда состояние меняется **не одним скаляром-полем, а вызовом конкретного метода** (швартовка:
`Unmoor()`, `MoorTo(dock)`, `ChangeRopeLength(x)`), не нужно ни опрашивать состояние каждый кадр
(ненадёжно), ни проигрывать клик кнопки (триггеры обманчивы — R10). **Перехватывай сам метод
Harmony-постфиксом и ретранслируй действие через хост.** Это эталонный приём для подобных механик.

### 5.1. Схема

```
ЛОКАЛЬНОЕ действие (игрок) → игровой метод X() отрабатывает локально
   → Harmony-postfix X (если НЕ _applying):
        client → шлёт RequestMsg хосту
        host   → шлёт StateMsg всем (broadcast/RelayExcept)
УДАЛЁННОЕ применение:
   host получил Request → Apply(...) (под _applying) + RelayExcept остальным
   client получил State → Apply(...) (под _applying)
Apply(): _applying=true; <воспроизвести эффект>; _applying=false   // finally!
```
Хост — авторитет (его пружина держит «настоящую» лодку; у клиента она кинематическая, швартовка
косметична, но отвязка клиента ОБЯЗАНА дойти до хоста, чтобы лодка освободилась). См. R3.

### 5.2. Правила паттерна (R11–R16)

- **R11. Перехватывай метод состояния, а не кнопку и не опрос.** Найди в декомпиле метод, который
  реально меняет состояние (`Unmoor`/`MoorTo`/`ChangeRopeLength`), и патчь именно его — он ловит ВСЕ
  пути вызова (pickup, триггер дока, `UnmoorAllRopes`, скролл) одной точкой.

- **R12. Harmony-postfix ОБЯЗАН быть в try/catch.** Исключение из постфикса пробрасывается в игровой
  поток и ломает механику (был баг: «канат завис в воздухе», отвязка срывалась на середине). Любой
  постфикс глотает свои ошибки и логирует. Это критично, не опционально.

- **R13. Защита от эха `_applying`.** Удалённое применение часто зовёт тот же игровой метод (или его
  эффект ловится тем же постфиксом). Ставь статический флаг `_applying` на время `Apply()` (в
  `finally`!), и постфикс при нём — ранний выход. Иначе бесконечный пинг-понг.
  Альтернатива: на приёме менять состояние «в обход» (см. R15) — тогда постфикс вообще не сработает.

- **R14. Стабильный индекс объекта, не зависящий от иерархии.** Адресуй объект индексом в его
  родном массиве (`BoatMooringRopes.ropes`), а не по `GetComponentInParent`: метод может
  **перепарентить** объект (после `MoorTo` трос висит под ПРИЧАЛОМ, не под лодкой). Строй карту
  `объект→индекс` заранее из массива; иерархия — только фолбэк.

- **R15. На приёме воспроизводи ПОЛНЫЙ эффект, а состояние STATE-значений ставь напрямую (абсолют).**
  - Игровой метод часто делает не всё: `Unmoor()` только репарентит — нужно ещё `ResetRopePos()`,
    иначе объект зависает. Сверяйся с тем, как игра делает это в комплексе (`UnmoorAllRopes` =
    `Unmoor`+`ResetRopePos`).
  - Для STATE-значения (длина) шли **абсолют** и ставь поле **напрямую**, а не вызывай игровой метод
    повторно: `ChangeRopeLength` нелинеен (`change*|change|`) и накапливал бы рассинхрон; прямая
    запись поля ещё и не триггерит постфикс (бесплатная защита от эха, ср. R13). Приватные части
    (`mooredToSpring`) бери рефлексией.

- **R16. Цели-объекты вне лодки — по real-space позиции.** Причал/остров/динги адресуй стабильной
  координатой (`CoordSpace.LocalToReal(target.position)`); приёмник находит ближайший объект нужного
  типа в радиусе (объекты мира статичны и одинаковы из одного сейва). NetId-реестр мира — позже.

### 5.3. Когда какой механизм (сводка выбора)

| Признак механики | Механизм | Пример |
|---|---|---|
| Непрерывный скаляр-поле, прямой ввод | STATE-синк значения + окно владения | лебёдка `currentLength` |
| Команда → физика (расходится у клиента) | команда client→host + визуал host→client | штурвал `currentInput` |
| Состояние меняется ВЫЗОВОМ метода | **реле-через-хост по перехвату метода (Часть 5)** | швартовка `Unmoor/MoorTo/ChangeRopeLength` |
| Простой дискретный клик-тоггл кнопки | форвард `ControlEvent` → replay обработчика | палубные кнопки |
| Поза физ-объекта у кинематического клиента | кинематик-кукла + NetTransform (кадр boat/real) | якорь, лодка |
